## 1. 한 문장 요약

한국 공공·기업의 HWPX 문서를 웹에서 열고, **Claude Sonnet 4와 대화하며 구조적으로 편집**한 뒤, 모든 변경을 단어 단위로 검토·되돌릴 수 있는 모던 뷰어/에디터.

## 2. 문제 정의

### 2.1 시장의 공백

한국 공공·법무·의료 분야에서 HWPX는 사실상의 표준 문서 형식이지만:

- **데스크톱 한컴오피스에 종속** — 라이선스 비용, 설치 강제, OS 종속(Windows 우세)
- **서버·웹 환경에서 프로그래밍적으로 다룰 수 있는 공식 SDK 부재** — pyhwpxlib 등 오픈소스가 일부 채우고 있으나 viewer/editor 레이어는 여전히 빈 곳
- **AI 협업 부재** — 워드/구글독스/노션은 LLM 통합이 활발한데, HWPX는 거의 없음

### 2.2 사용자 고통점

- 정부 양식·계약서·보고서를 매번 한컴에서 열어 수정 → 클라우드 협업 어려움
- 비슷한 양식 여러 개를 일괄 채우는 작업이 수작업
- 문장을 더 격식 있게, 또는 일관된 톤으로 다시 쓰는 작업에 LLM 활용하지 못하고 복붙으로 처리
- 누가 어디를 언제 바꿨는지 추적 어려움

## 4. 제품 원칙

순서가 있는 우선순위입니다. 충돌 시 위가 이깁니다.

1. **AI co-editing이 일급 시민** — 채팅·인라인 편집·diff·cherry-pick은 부가 기능이 아니라 제품의 본질.
2. **변경의 가시성과 가역성** — 모든 편집은 이력에 남고, 언제든 되돌릴 수 있어야 한다. 사용자가 AI를 신뢰할 수 있게 하는 안전장치.
3. **격식 있는 한국어 공공 문서 톤** — UI는 모던하되, 산출물은 한국 공공/기업 양식의 격식을 존중.
4. **즉각적 피드백** — 스트리밍 응답, hover 상태, flash 하이라이트로 시스템이 살아있다는 감각.
5. **단순함 > 풍부함** — MVP에서는 표면적이 작더라도 핵심 흐름이 매끄럽게.

## 5. 핵심 User Story

각 스토리는 우선순위(P0/P1/P2)와 현재 구현 상태(✅/⚠️/⬜)를 표기합니다.

### P0 — MVP 필수

- ✅ **US-1**: 사용자로서, 문서를 페이지 단위로 보고 무한 스크롤하면서 전체를 훑고 싶다.
- ✅ **US-2**: 사용자로서, 사이드바에서 페이지 미니어처를 클릭해 해당 페이지로 이동하고 싶다.
- ✅ **US-3**: 사용자로서, AI 채팅에 자연어로 "예산 표를 정리해줘" 같은 요청을 보내고 싶다.
- ✅ **US-4**: 사용자로서, AI가 적용한 변경을 단어 단위 diff로 검토하고 싶다.
- ✅ **US-5**: 사용자로서, AI 응답을 보면서 "되돌리기"로 즉시 원상복구할 수 있어야 한다.
- ✅ **US-6**: 사용자로서, 본문 텍스트를 드래그해 "다시 쓰기/간결하게/영어로" 같은 부분 편집을 트리거하고 싶다.
- ✅ **US-7**: 사용자로서, AI 제안 중 일부 변경만 받아들이고 싶다 (cherry-pick).
- ✅ **US-8**: 사용자로서, AI와 멀티턴 대화하면서 이전 맥락을 잃지 않고 후속 요청을 하고 싶다.
- ⚠️ **US-9**: 사용자로서, 실제 .hwpx 파일을 업로드하고 편집한 결과를 다시 .hwpx로 다운로드하고 싶다.
  → 현재 mock 데이터만. pyhwpxlib 연결 필요.
- ⬜ **US-10**: 사용자로서, 변경 이력 전체 타임라인을 한눈에 보고, 각 시점으로 토글하고 싶다. (현재 부분 구현, 패널은 있으나 일부 액션 미완)

### P1 — MVP+

- ⚠️ **US-11**: 사용자로서, 표의 셀 단위로 변경을 부분 수락하고 싶다. (현재 시각화만)
- ⬜ **US-12**: 사용자로서, 문서 내 텍스트를 검색하고 결과로 점프하고 싶다. (UI는 있으나 동작 안 함)
- ⬜ **US-13**: 사용자로서, 다른 사람과 같은 문서를 동시에 보고 편집하고 싶다 (presence + collab cursor).
- ⬜ **US-14**: 사용자로서, 자주 쓰는 양식(정부 R&D 계획서 등)을 템플릿으로 저장·재사용하고 싶다.
- ⬜ **US-15**: 관리자로서, 조직의 모든 문서 편집 활동을 감사 로그로 받고 싶다.

### P2 — 후순위

- ⬜ **US-16**: 사용자로서, 모바일에서도 읽기 전용으로 문서를 보고 싶다.
- ⬜ **US-17**: 사용자로서, 음성으로 "이 페이지 결론을 더 간결하게"라고 말하고 싶다.
- ⬜ **US-18**: 개발자로서, 이 뷰어를 우리 SaaS에 iframe/웹 컴포넌트로 임베드하고 싶다.

## 7. 비기능 요구사항

| 항목 | 목표 |
|---|---|
| **첫 화면 표시** (LCP) | 1.5s 이하 (캐시 후) |
| **AI 응답 첫 토큰까지** (TTFB) | 3s 이하 |
| **단어 단위 diff 계산** | 5,000자 문서에서 100ms 이하 |
| **무한 스크롤 60fps** | 50 페이지까지 끊김 없이 |
| **메모리** | 50번 편집 후 추가 메모리 사용량 < 50MB |
| **접근성** | 키보드 내비게이션 가능 (모든 핵심 액션), focus ring, ARIA labels |
| **브라우저 지원** | 최신 Chrome / Edge / Safari / Firefox 2개 메이저 버전 |
| **다국어** | UI는 한/영. 콘텐츠는 한국어 우선, 영어 부분 번역 액션 지원. |
| **보안** | API 키는 서버측에서만 (claude.ai Artifacts에서는 자동 처리). 사용자 문서는 로컬 처리 우선. |

## 8. 성공 지표 (제안)

### 도입 지표
- 주간 활성 문서 편집자 수 (WAU)
- 세션당 평균 편집 횟수
- AI 채팅 사용률 (편집의 몇 %가 AI 경유인가)

### 품질 지표
- AI 편집 적용률 (제안 → 실제 적용)
- 되돌리기율 (적용 → 즉시 되돌림 — 높으면 AI 품질 문제)
- Cherry-pick 사용률 (전체 수락 vs 부분 수락 비율)

### 비즈니스 지표 (Phase 3+, 모네타이제이션 단계)
- 무료 → 유료 전환율
- 조직 단위 도입 (B2B 라이선스)
- API/임베드 사용량

## 10. 비즈니스 모델 (요약)

상세는 별도 문서. 여기서는 제품 결정에 영향을 주는 부분만:

- **Phase 1 (현재)**: 무료, 최대 도입 우선
- **Phase 2**: 익명 사용량 텔레메트리 (~100명 시점)
- **Phase 3**: B2B 유료 — 연간 라이선스, SI 지원, 커스텀 양식, 온프레미스/프라이빗 배포

제품 결정 영향:
- **온프레미스 배포 가능성을 항상 염두**. 즉, 외부 의존을 최소화. (Artifacts에서 fetch 외 외부 호출 안 됨)
- **공공/의료/법무를 타겟하므로 데이터 외부 전송 옵션을 명확히 분리**할 수 있어야 함. 향후 WASM 클라이언트 사이드 옵션 고려.
- **사용량 기반 가격 책정 시 cherry-pick 등 fine-grained 액션도 카운트 가능해야 함** — 이벤트 추적 훅 위치 미리 마련.

## 11. 위험과 대응

| 위험 | 영향 | 대응 |
|---|---|---|
| HWPX 사양이 비공식적·복잡함 | 파싱 실패 | pyhwpxlib에 의존, 변환 레이어 분리. 뷰어는 블록 모델만 다룸. |
| LLM이 JSON 형식 깨뜨림 | 편집 적용 실패 | `extractJson` 두 단계 fallback, 실패 시 사용자에게 명시적 에러. 시스템 프롬프트 강화. |
| 토큰 비용 폭증 (멀티턴) | 단가 부담 | 대화 N turn 후 자동 요약. 옵션: 페이지 컨텍스트 일부만 전송. |
| 한컴이 자체 AI 통합 출시 | 차별점 약화 | 모네타이제이션을 빠르게: 공공 양식 100종 라이브러리, SI 벤더 직접 영업으로 lock-in. |
| 사용자가 AI 편집을 신뢰하지 않음 | 도입 정체 | diff·cherry-pick·history·undo로 안전장치 가시화. "AI를 멀리하지 않고 가까이서 통제" UX. |

## 13. 결정 기록 (Decision Log)

향후 결정이 추가되면 이 섹션에 적층.

- **2026-04**: 첫 버전은 단일 React 컴포넌트로. 프로덕션 이전 시 분리 예정. (이유: Artifacts 단일 파일 제약 + 빠른 반복)
- **2026-04**: history는 entry 삭제 없이 `undone` 토글 모델. (이유: 단순성, 멀티 시점 토글 필요)
- **2026-04**: 시스템 프롬프트는 JSON 응답 강제. (이유: 파싱 안정성. 트레이드오프: 약간의 자연스러움 손실)
- **2026-04**: 멀티턴은 첫 turn에만 문서 컨텍스트, 이후 turn은 짧은 사용자 입력. (이유: 토큰 절약)
- **2026-04**: 데스크톱 우선 (사이드바·dock·우측 패널 동시 노출). 모바일 v1에서 제외.
